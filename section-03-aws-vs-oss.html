Decision principles for how I would build.

I would make infrastructure choices based on product speed, operational simplicity, privacy boundaries, and long-term portability, not on ideology.

ConsiderationManaged biasPortable biasWhat I would optimize for
China deployment AWS S3 doesn't work in mainland China. SageMaker has minimal China presence. Requires a separate Chinese AWS account with different APIs and legal entity. MinIO self-hosts on Alibaba Cloud or Huawei Cloud. Same S3-compatible API, same Terraform module, different provider. vLLM runs on both platforms' GPU instances. Hard requirement
Data sovereignty (PIPL) Complex compliance posture. AWS China regions exist but are separate legal entities with unclear PIPL status. Self-hosted MinIO in Chinese cloud: data physically never leaves China. Clean legal position. Hard requirement
Schema evolution for Higg updates RDS/Redshift schema changes require migrations. Breaking changes for downstream systems every time methodology updates. Iceberg schema evolution: add columns in place, zero downtime, historical data stays queryable at old schema. Annual requirement
LLM data privacy SageMaker managed inference endpoints. Supplier chemical inventory and social audit data cannot go to managed US endpoints. vLLM self-hosted. Supplier data never leaves Worldly infrastructure. Clean under data agreements. Data agreements
Cost at scale Managed services usually trade higher unit cost for faster setup and lower operational load. Portable components usually lower long-run cost, but they ask more of the engineering team. Tradeoff
Setup speed Faster initial configuration. Less infrastructure management. Better if team size is the constraint. More decisions upfront. More infrastructure to manage. Slower to first pipeline. Tradeoff
My bias
Managed where it reduces drag. Portable where it protects the product. I would use managed infrastructure for control-plane convenience, but keep the core data and model-serving layers portable where privacy, deployment flexibility, and long-lived product constraints matter more.
How I would think about the tradeoff, qualitatively.
Managed-first
SageMaker + Glue + Redshift + MSK
Fastest start
Good when the priority is speed to a first operating baseline and the constraints are mostly inside one cloud. Less attractive when regional portability and private model serving become core product needs.
Recommended
Hybrid operating model
EKS + MinIO + Iceberg + OSS inference
Best balance
This is the operating model I would start with when the business needs both delivery speed and long-term flexibility. It keeps the team moving while avoiding hard coupling in the layers most likely to matter later.
Self-managed everywhere
Self-managed K8s + MinIO + everything OSS
Highest control
I would avoid this early unless there is a very strong constraint forcing it. It creates the most operational burden at exactly the point where the team should be learning fast and keeping execution calm.
The point is not to predict exact spend. The point is to choose an operating model that matches product constraints, team capacity, and the pace of iteration you want.
